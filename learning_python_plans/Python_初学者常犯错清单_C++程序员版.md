# Python 初学者常犯错清单（C++ 程序员版）

> 适用对象：有较强 C++ 背景，但刚开始系统学习 Python 的开发者。  
> 阅读方式：每条先看“错误直觉”，再看“为什么容易踩坑”，最后看“正确建议”。

---

## 1. 误以为赋值等于复制

### 错误直觉

在 C++ 里，写下赋值语句时，经常会自然联想到值拷贝。

### Python 中的坑

```python
a = [1, 2, 3]
b = a
b.append(4)

print(a)  # [1, 2, 3, 4]
```

### 为什么容易踩坑

Python 中变量更像“名字绑定到对象”，不是“盒子里装着值”。

### 正确建议

- 要复制列表时，用 `a.copy()` 或切片 `a[:]`
- 分清：
  - `==` 比较值
  - `is` 比较是不是同一个对象

## 2. 把可变对象作为默认参数

### 错误写法

```python
def add_item(x, items=[]):
    items.append(x)
    return items
```

### 问题表现

```python
print(add_item(1))  # [1]
print(add_item(2))  # [1, 2]
```

### 为什么容易踩坑

C++ 程序员会下意识把它理解成“每次调用时新建一个默认容器”，但 Python 默认参数只在函数定义时创建一次。

### 正确写法

```python
def add_item(x, items=None):
    if items is None:
        items = []
    items.append(x)
    return items
```

## 3. 过度依赖下标循环

### 错误直觉

```python
nums = [10, 20, 30]
for i in range(len(nums)):
    print(i, nums[i])
```

### 为什么不够 Pythonic

这不是错，但常常不是最自然的 Python 写法。

### 更推荐

```python
for i, x in enumerate(nums):
    print(i, x)
```

### 正确建议

- 直接遍历元素优先
- 需要索引时用 `enumerate`
- 多序列并行遍历用 `zip`

---

## 4. 把 Python 写成“没类型声明的 C++”

### 常见表现

- 写很多不必要的类层次
- 把简单脚本问题都包进复杂对象模型
- 强行模仿 C++ 的工程结构

### 为什么会这样

因为你已经习惯了大型 C++ 工程的严谨性，但 Python 的优势在于表达力和简洁性。

### 正确建议

先问自己：

- 这个问题用函数能解决吗？
- 这个数据结构用 `dict` 或 `dataclass` 能解决吗？
- 我是不是为了“显得专业”而引入了不必要复杂度？

## 5. 误用 `is` 来比较值

### 错误写法

```python
x = 1000
y = 1000

if x is y:
    print("equal")
```

### 为什么危险

`is` 比较的是对象身份，不是值相等。某些小整数或短字符串看起来“正好能用”，只是实现细节，不应依赖。

### 正确写法

```python
if x == y:
    print("equal")
```

### 正确建议

- 比较值：`==`
- 比较是否同一个对象：`is`
- 最常见的 `is` 场景是：`if x is None:`

## 6. 误以为 `tuple` 只是“不能改的 list”

### 错误理解

把 `tuple` 简单当作只读列表。

### 更准确的理解

`tuple` 更适合作为：

- 固定结构的数据返回值
- 不希望被修改的轻量记录
- 解包对象

### 示例

```python
def minmax(a: int, b: int) -> tuple[int, int]:
    return (a, b) if a < b else (b, a)
```

### 正确建议

把 `tuple` 想成“固定结构记录”，而不是单纯的“不能改 list”。

## 7. 不习惯推导式，结果代码很啰嗦

### 常见写法

```python
even = []
for x in nums:
    if x % 2 == 0:
        even.append(x)
```

### Python 更常见写法

```python
even = [x for x in nums if x % 2 == 0]
```

### 为什么值得学

推导式是 Python 的日常主力写法，不是高级技巧。

### 正确建议

- 简单过滤/映射优先想到推导式
- 过于复杂时再拆回普通循环

## 8. 不愿意用标准库，凡事手搓

### 错误直觉

“我自己写也不难。”

### Python 里的现实

很多你想手写的功能，标准库里已经有了：

- `pathlib`
- `collections.Counter`
- `itertools`
- `functools`
- `json`
- `csv`
- `argparse`

### 正确建议

先查标准库，再决定要不要自己写。  
Python 的生态价值，很大一部分就在“少造轮子”。

## 9. 不重视 `with`

### 错误写法

```python
f = open("data.txt", "r", encoding="utf-8")
text = f.read()
f.close()
```

### 为什么不推荐

- 容易遗漏关闭
- 出异常时资源释放不稳定

### 正确写法

```python
with open("data.txt", "r", encoding="utf-8") as f:
    text = f.read()
```

### 正确建议

把 `with` 当成 Python 版通用 RAII 入口：

- 文件
- 锁
- 网络连接
- 数据库连接
- 临时资源

都优先考虑它。

---

## 10. 误把 `__del__` 当析构函数依赖

### 错误直觉

C++ 程序员容易自然联想到析构时机是可靠的。

### Python 中的问题

`__del__` 调用时机并不适合承担关键资源管理责任。

### 正确建议

- 资源管理优先使用 `with`
- 需要清理动作时，优先显式写关闭逻辑或上下文管理器
- 不要把 `__del__` 当作 C++ 析构函数来依赖

---

## 11. 过早焦虑性能

### 常见表现

- 一上来就担心 Python 慢
- 先做微优化，再写功能
- 想把 Python 写得尽量像 C++

### Python 中更好的顺序

1. 先写对
2. 先写清楚
3. 先测量
4. 再优化热点

### 正确建议

Python 优化的优先路线通常是：

- 使用内建函数
- 使用标准库
- 使用 `numpy` / `pandas`
- 把性能热点下沉到 C++/Rust 扩展

## 12. 把所有东西都设计成类

### 错误直觉

“面向对象更专业。”

### Python 中常见更优方案

很多问题只需要：

- 函数
- 模块
- `dict`
- `dataclass`

### 正确建议

如果没有明显状态封装需求，不必急着上类。  
Python 中模块级函数和轻量数据结构完全可以非常专业。

## 13. 忽略列表、字典的浅拷贝与深拷贝区别

### 示例

```python
a = [[1, 2], [3, 4]]
b = a.copy()
b[0].append(99)

print(a)  # [[1, 2, 99], [3, 4]]
```

### 为什么会出问题

`copy()` 对外层容器做的是浅拷贝，内部元素仍然共享引用。

### 正确建议

- 理解浅拷贝与深拷贝的区别
- 复杂嵌套结构需要时使用 `copy.deepcopy()`

---

## 14. 不习惯异常机制，想用返回码兜底

### C++ 习惯迁移问题

你可能更熟悉：

- 错误码
- 返回状态
- 输出参数

### Python 常见方式

Python 更常使用异常表达失败。

```python
def divide(a: float, b: float) -> float:
    if b == 0:
        raise ValueError("b cannot be 0")
    return a / b
```

### 正确建议

- 失败场景优先用异常机制表达
- 捕获异常时尽量精确，不要一网打尽

---

## 15. 写代码不分模块，所有内容堆一个文件

### 常见问题

初学 Python 容易把所有函数、逻辑、测试都堆在一个 `.py` 文件里。

### 正确建议

尽早建立这些习惯：

- 功能拆函数
- 文件拆模块
- 测试独立出来
- 公共逻辑不要散落复制

### 一个简单结构示意

```text
project/
├── src/
│   └── app.py
├── tests/
│   └── test_app.py
└── pyproject.toml
```

## 16. 不写类型注解，后期自己也看不懂

### 错误直觉

“Python 是动态语言，不用写类型。”

### 问题

项目稍微一复杂，接口含义不清、重构困难、IDE 提示也变差。

### 推荐做法

```python
def top_words(text: str, top_n: int) -> list[tuple[str, int]]:
    ...
```

### 正确建议

- 公共函数优先加类型注解
- 返回复杂结构时更要写清楚
- 不必一开始 100% 全标注，但应尽早养成习惯

---

## 17. 误把列表推导式写得过于复杂

### 问题示例

```python
result = [x * 2 for x in nums if x > 0 if x % 2 == 0 if x < 100]
```

### 问题

虽然语法合法，但可读性开始下降。

### 正确建议

- 简单逻辑用推导式
- 复杂逻辑拆成普通循环或辅助函数
- Pythonic 的前提是清晰，不是越短越好

## 18. 手工拼路径字符串

### 不推荐

```python
path = base_dir + "/data/" + filename
```

### 推荐

```python
from pathlib import Path

path = Path(base_dir) / "data" / filename
```

### 正确建议

现代 Python 里路径处理优先使用 `pathlib`，尤其在跨平台场景下更稳妥。

## 19. 忽视虚拟环境

### 常见问题

- 全局乱装包
- 项目之间依赖冲突
- 过段时间自己也不知道装过什么

### 正确建议

每个项目尽量使用独立虚拟环境。  
即便是小项目，也要尽早建立这个习惯。

---

## 20. 不写测试，靠手工点点点验证

### 常见误区

“脚本不大，不用测试。”

### 为什么这是坑

一旦你开始重构或扩展，手工验证很快就不可靠了。

### 正确建议

尽早接触 `pytest`：

- 关键函数写最小测试
- 先保护核心逻辑
- 这样改代码时更安心

---

## 21. 误解并发：以为开线程就能解决 CPU 密集性能问题

### C++ 直觉

多线程常常意味着更直接的并行能力。

### Python 现实

受 GIL 影响，CPU 密集任务通常不靠普通线程获得理想加速。

### 正确建议

- I/O 密集：`threading` / `asyncio`
- CPU 密集：`multiprocessing`、`numpy`、C/C++ 扩展

---

## 22. 学语法很多，却不练真实小项目

### 常见问题

- 看了很多教程
- 记住很多零散语法
- 真要写脚本时还是发懵

### 正确建议

尽早做小项目，比如：

1. 文件批处理工具
2. 文本词频统计器
3. CSV 数据清洗脚本
4. 目录统计工具

写项目时学到的 Python，才更容易变成真正的直觉。

---

## 23. 只追求“短”，忽略可读性

### 误区

觉得 Python 代码越短越高级。

### 正确建议

Python 的核心不是“越短越好”，而是：

- 更清楚
- 更自然
- 更容易维护

如果一行写法让两天后的你都皱眉，那它多半就不够好。

---

## 24. 忽略文档与命名质量

### 常见问题

- 变量命名模糊
- 函数职责不清
- 参数意义要靠猜

### 正确建议

- 变量名尽量表达意图
- 函数做一件事
- 需要时写简短 docstring
- 类型注解和命名一起提升可读性

---

## 给 C++ 程序员的 8 条避坑口诀

1. 赋值是绑定，不一定是复制
2. 可变默认参数千万别乱用
3. `==` 比值，`is` 比身份
4. 迭代优先，少写下标循环
5. `with` 优先，不依赖 `__del__`
6. 先用函数和标准库，别急着上复杂类设计
7. 先写清楚，再优化
8. 多练小项目，少囤纯语法知识
